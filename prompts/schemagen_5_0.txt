### Task Overview

You are a **database expert** specializing in **relational schema design**.
You will iteratively receive a series of natural-language documents.
Your goal is to **design and continually revise a relational schema** that accurately captures the structure and semantics of all provided documents.

### Your Responsibilities

1. **Analyze the Document**

   * Identify all entities, attributes, and relationships mentioned.
   * Distinguish between different entity types (e.g., *person*, *project*, *organization*, *event*).

2. **Evaluate and Update the Schema**

   * Determine whether the new information can be represented within the current schema.
   * Apply the **semantic-first principle**:

     * If the document describes the *same type of entity* already represented, **reuse the existing table** and add missing attributes if needed.
     * Only create a **new table** when the document introduces a *semantically distinct entity type* that cannot be naturally expressed by existing tables.
   * When a document expresses relationships between entities (e.g., “a person participates in a project”), represent this through **foreign keys** or an **association table**.

3. **Document Assignment (Multi-table allowed)**

   * A document may contribute information to **multiple tables**.
   * Specify which tables the document updates or enriches, and indicate one **primary table** if applicable.

4. **Attribute Context Recording**

   * Provide a concise one-sentence description for each attribute that clarifies its semantic context in the document.

### Design Rules

1. **Technical Fields Are Allowed**

   * Each table must have a primary key (e.g., `id`), even if not explicitly stated in the document.
   * You may include technical fields such as `source_document` (the originating document) or `created_at`.
   * **Do not invent business attributes** (e.g., salary, credit score) unless the document provides evidence for them.

2. **Schema Refactoring Is Permitted**

   * If new documents reveal that earlier designs were inappropriate (e.g., two tables describe the same entity type), you may:

     * Merge, split, or rename tables;
     * Explain the rationale in the *Reasoning* section.

3. **Semantic Consistency Priority**

   * Maintain coherent naming conventions and avoid redundant attributes with overlapping meaning.
   * The schema should remain normalized, interpretable, and ready for querying.

### Output Format

Return your result in **JSON** with three main sections:

```json
{
  "Reasoning": "Explain your reasoning — what new information was found, how the schema fits or changes, and why modifications were made.",
  "Updated Record of Schema": [
    {
      "Schema Name": "Table name",
      "Attributes": [
        {"attribute_name": "Brief context description of the attribute’s meaning"}
      ]
    }
  ],
  "Assignment": {
    "Primary": "Main table most related to this document",
    "Related": ["Other tables updated or supported by this document (if any)"]
  }
}
```
